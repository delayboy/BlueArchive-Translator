# CharacterDialogEventExcel 打包前后 bytes 不一致问题分析与修复方案

## Context

用户在开发蔚蓝档案汉化补丁时，发现 `CharacterDialogEventExcel` 表通过 `process_excel.py` 打包后放到游戏里会导致文本加载不出来，而其他表都正常。通过 `flatbuffer单测.py` 发现：即使不修改 JSON，打包前后的 bytes 流也不一样，但解析出的 JSON 却相同。需要分析 bytes 变化的原因，以及这是否是导致游戏文本加载失败的根因。

## 根因分析

### 1. 为什么 bytes 打包前后会变化

通过解码 `test_bytes` 的 vtable 结构，并对照 `pack_CharacterDialogEventExcel` 的实现，确认了以下关键事实：

**原始 bytes 的 vtable 结构（offset 6 开始，size=58 字节 = 4 + 27×2）：**
- 共 27 个 field slot（字段 0-26）
- 字段 20（VoiceId）的 vtable offset = **0**（即字段不存在/被省略）
- 字段 27（ScenarioCharacterShapes）根本不在 vtable 中（原始数据用旧版 schema 生成，只有 27 个字段）

**打包后的 bytes 结构：**
- vtable 经过 `WriteVtable` 的 trailing-zero trim 逻辑（`builder.py:258-274`）后，也是 27 个 slot（因为字段 27 是默认值 0 被裁剪），**vtable 大小相同**
- 但**字段 20（VoiceId）的 vtable offset 变成非 0**（指向一个空向量）

#### 核心差异：VoiceId 字段从「不存在」变成了「存在但为空向量」

这是 `repack_wrapper.py:4788-4794` 的实现缺陷：

```python
VoiceId_vec = 0
if 'VoiceId' in data:          # ← dump 总会输出 "VoiceId": []，所以恒为 True
    VoiceId_items = data['VoiceId']
    CharacterDialogEventExcel.StartVoiceIdVector(builder, len(VoiceId_items))  # len=0
    for item in reversed(VoiceId_items):   # 空列表，不执行
        builder.PrependUint32(convert_uint(item, password))
    VoiceId_vec = builder.EndVector(len(VoiceId_items))   # ← 返回非 0 偏移！
```

`builder.EndVector(0)` 即使对空向量也会返回一个**非 0 的偏移**（`builder.py:443-458`，它会 `PlaceUOffsetT(0)` 并返回当前 offset）。于是 `AddVoiceId(builder, 非零偏移)` 调用 `PrependUOffsetTRelativeSlot(20, 非零, 0)`，由于 `非零 != 0`（默认值），字段被写入 vtable（`builder.py:691-700`）。

而 flatbuffers 的 `PrependSlot`/`PrependUOffsetTRelativeSlot` 只在 `value == default` 时才跳过字段（`builder.py:646-653, 691-700`）。空向量有非零偏移，所以不会被跳过。

**原始数据中 VoiceId 为空时直接省略该字段（vtable offset=0），但打包后变成「存在且指向空向量」** —— 这就是 bytes 变化的主因。它会导致：
- vtable 中字段 20 的 slot 从 0 变成非 0（2 字节内容变化）
- table data 中多出 4 字节的向量偏移
- buffer 中多出 4 字节的空向量（长度前缀=0）
- 后续所有偏移和 padding 发生位移

#### 次要差异：空字符串字段（对其他记录可能生效）

`builder.CreateString("")` 同样返回非 0 偏移（`builder.py:477-501`），所以空字符串字段也会被写入 vtable。如果原始数据中某些字符串字段为空时被省略，打包后也会出现「不存在→存在但为空」的变化。本测试用例中所有字符串都非空，所以未触发，但表中其他记录可能命中。

#### 非差异项（已排除）

- **vtable slot 数量**：虽然 schema 有 28 个字段，但 `WriteVtable` 会裁剪尾部 0 slot。字段 27 值为 0 被裁剪，最终 vtable 仍是 27 个 slot、58 字节，与原始一致。
- **标量字段默认值**：`PrependInt32Slot(slot, 0, 0)` 等在值等于默认值时跳过，与原始行为一致。
- **加密**：测试用 `encrypt=False`（password=None），实际 DB repack 流程也是 `encrypt=False`（`table.py:365`），无加密差异。

### 2. 这个变化是否可能导致游戏文本加载失败

**有可能，而且可能性较大。** 关键在于 `VoiceId` 字段的「存在性」发生了语义变化，而非仅仅是字节布局变化：

`CharacterDialogEventExcel.py:179-181` 提供了 `VoiceIdIsNone()` 方法：
```python
def VoiceIdIsNone(self):
    o = flatbuffers.number_types.UOffsetTFlags.py_type(self._tab.Offset(44))
    return o == 0
```

- 原始数据：`VoiceIdIsNone()` 返回 `True`
- 打包后数据：`VoiceIdIsNone()` 返回 `False`（字段存在，指向空向量）

如果游戏端（C# / IL2CPP）的对话加载逻辑用类似 `HasVoiceId` / `VoiceIdIsNone` 来分支处理 —— 例如「有 VoiceId 时走语音+字幕同步加载流程，没有时走纯文本流程」—— 那么打包后数据会让游戏误以为有语音数据，进入语音加载分支后遇到空向量可能抛异常或返回空，导致整条对话文本也加载不出来。这能解释为什么只有 `CharacterDialogEventExcel` 出问题（它是少数有 VoiceId 向量字段且游戏可能按存在性分支处理的表）。

**但需注意**：仅凭静态分析无法 100% 确认这就是游戏端失败点，需要通过修复后实测验证。

## 修复方案

### 主修复：VoiceId 为空时不创建向量

修改 `FlatData/repack_wrapper.py:4788-4794`（`pack_CharacterDialogEventExcel` 函数内）：

```python
VoiceId_vec = 0
if data.get('VoiceId'):  # 仅在非空时创建向量，空列表/None 均跳过
    VoiceId_items = data['VoiceId']
    CharacterDialogEventExcel.StartVoiceIdVector(builder, len(VoiceId_items))
    for item in reversed(VoiceId_items):
        builder.PrependUint32(convert_uint(item, password))
    VoiceId_vec = builder.EndVector(len(VoiceId_items))
```

修复后：VoiceId 为空时 `VoiceId_vec` 保持 0，`AddVoiceId(builder, 0)` 触发 `PrependUOffsetTRelativeSlot(20, 0, 0)`，因 `0 == 0`（默认值）字段被跳过，与原始数据行为一致。

### 次修复：空字符串字段同样跳过（建议同步修复）

对 `pack_CharacterDialogEventExcel` 中所有字符串字段采用相同模式（`repack_wrapper.py:4780-4787`），例如：

```python
ActionName_off = builder.CreateString(encrypt_string(data['ActionName'], password)) if data.get('ActionName') else 0
```

应用到 ActionName、AnimationName、LocalizeKR、LocalizeJP、LocalizeTH、LocalizeTW、LocalizeEN、LocalizeCVGroup 共 8 个字符串字段。这样其他记录中如果某项字符串为空（例如 LocalizeCVGroup 经常为空），也不会出现「不存在→存在但空」的语义变化。
