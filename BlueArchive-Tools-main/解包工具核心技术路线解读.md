## 结论：**典型的"解包 → 修改 → 重打包"工作流**

本项目 **不是** HOOK 注入、**不是** 网络代理实时翻译、**不是** 内存补丁。它是一套**离线资源翻译工具链**：从官方 CDN 拉取资源 → 解密解码为可编辑格式 → 人工翻译后 → 重新加密打包回去 → 绕过游戏完整性校验。

## 三条主要数据通路

### 1️⃣ 剧情/数据表汉化（Excel.zip / ExcelDB.db）

入口：[process_excel.py](vscode-webview://0jekke33biiooagsj8e6cokn7mg180focrdpc13ob6nks03d881e/process_excel.py) → [xtractor/table.py](vscode-webview://0jekke33biiooagsj8e6cokn7mg180focrdpc13ob6nks03d881e/xtractor/table.py)

- **Excel.zip**：ZIP 加密压缩包，里面是 **FlatBuffer `.bytes`** 文件。文件经 XOR 加密（密钥 = FlatBuffer 类名）。
- **ExcelDB.db**：**SQLCipher** 加密的 SQLite 数据库，`Bytes` 列里存 FlatBuffer 二进制。
- 解包流程：解密 → 用 [BA-FlatData](https://github.com/beichen23333/BA-FlatData.git) 提供的 FlatBuffer Schema 反序列化为 JSON（可读，便于翻译）→ 翻译后用 `pack_*` 函数重新序列化 → 再 XOR 加密 → 重新压回 ZIP / 写回 SQLite → `VACUUM` 优化。

### 2️⃣ Unity AssetBundle 资源汉化（UI 图、字体、TextAsset）

入口：[xtractor/bundle.py](vscode-webview://0jekke33biiooagsj8e6cokn7mg180focrdpc13ob6nks03d881e/xtractor/bundle.py)

- 用 **UABEA CLI（H9per 版）** 拆解/导入 AssetBundle（Texture2D / Sprite / Font / TextAsset / MonoBehaviour 等）。
- 翻译后写回 bundle 时，调用 **[pycrcmanip](https://github.com/rr-/pycrcmanip)** 的 `apply_patch` 算法在文件末尾追加 4 字节，**强行让修改后文件的整体 CRC32 等于原值**（见 [bundle.py:262-314](vscode-webview://0jekke33biiooagsj8e6cokn7mg180focrdpc13ob6nks03d881e/xtractor/bundle.py#L262-L314)）。这样游戏读取到正确的 CRC，不会重新下载该资源。

### 3️⃣ 安卓客户端 APK 整体重打包

入口：[update/update_apk.py](vscode-webview://0jekke33biiooagsj8e6cokn7mg180focrdpc13ob6nks03d881e/update/update_apk.py) → [utils/apktools.py](vscode-webview://0jekke33biiooagsj8e6cokn7mg180focrdpc13ob6nks03d881e/utils/apktools.py)

- 从官方下载分体 APK（`UnityDataAssetPack` + 主包），用 **apktool** 反编译。
- 改包名实现共存版、改 `strings.xml`（`ブルアカ` → `蔚蓝档案`）、改 `SDKConfigSettings.json` 的 `Sdk_Url`、改 `gt4.js` 登录页语言。
- `GameMainConfig` 是 bundle 里的 XOR 加密 TextAsset，需要解开 → 改 JSON → 再 XOR 加密回去。
- apktool 重打包 → **zipalign** → **apksigner** 用自带 keystore (`beichen.jks`) 签名 → 产出 `蔚蓝档案.apk`。

## 绕过完整性校验的"骚操作"（核心思路所在）

游戏对汉化最大的两个障碍是 **加密** 和 **校验**，本项目分别用以下方法应对：

1. **加密**：原生支持 XOR(类名密钥) + Zip 密码 + SQLCipher，反向实现即可。

2. 校验绕过

   有两种策略并存：

   - **改 catalog 法**（用于 Table）：修改后重算 CRC 写回 `TableCatalog.bytes`，游戏比对 catalog 时认为合法。
   - **CRC 伪造法**（用于 Bundle）：不改 catalog，而是**让修改后的文件 CRC 仍等于原值** —— 通过 `pycrcmanip` 在文件尾追加 4 字节实现，这是这个项目里最巧妙的技术点。

## 一、核心发现：密钥派生链 = "公开字符串 → xxHash → 梅森旋转"

看 [lib/encryption.py:66-69](vscode-webview://0jekke33biiooagsj8e6cokn7mg180focrdpc13ob6nks03d881e/lib/encryption.py#L66-L69) 的 `create_key`：



```python
def create_key(name: str, size: int = 8) -> bytes:
    seed = calculate_hash(name)          # xxh32(name)
    return MersenneTwister(seed).next_bytes(size)
```

派生链是这样的：



```
[字符串]  ──xxh32──▶  [32位整数种子]  ──MT19937──▶  [N字节密钥流]  ──XOR──▶  密文
```

也就是说，输入只是一个**字符串**（类名、文件名、字段名），算法是**公开的标准算法**（xxHash + 梅森旋转 PRNG）。**整个加密强度等于零的"伪加密"**，知道算法的人立刻就能还原。

## 三、回答你的两个问题

### Q1：密钥从哪来？不逆向能拿到吗？

要分两类看：

| 加密类型                                   | 密钥来源                                   | 不逆向能拿到吗                                               |
| ------------------------------------------ | ------------------------------------------ | ------------------------------------------------------------ |
| FlatBuffer XOR / ZIP 密码 / GameMainConfig | **公开字符串派生**（类名、文件名、字段名） | ✅ **可以**。只要有人逆向过一次、把算法和 Schema 公开（就是 BA-FlatData 仓库），后面所有人都不用再逆向 |
| ExcelDB.db SQLCipher                       | **IL2CPP 二进制里的硬编码字符串**          | ❌ **不行**。这是真正的逆向产物，需要在 `libil2cpp.so` 里找到。本项目把它做成命令行参数让你自己填 |

**关键认知**：所谓"密钥"大部分时候不是一串神秘数字，而是一个**算法 + 一个公开输入字符串**。逆向的真正成本是搞清楚算法本身（一次性工作），而不是每次都去偷密钥。

### Q2：密钥是每版变化还是固定？

- **算法层（xxh32 + MT19937 + XOR）**：**永久固定**。这是 C# 代码里硬写的，几乎不会改。
- **派生输入字符串**（类名、字段名）：**几乎固定**。除非游戏开发商重命名内部数据表（极少发生），否则几个版本内不会动。
- **SQLCipher 数据库密钥**：**理论上可变**，但实际上 Yostar 历来没怎么换过。换一次意味着所有玩家都要重新下载资源，成本很高。如果哪天换了，汉化组就得重新去 IL2CPP 里挖一次。
- **签名密钥**（[apktools.py:40-44](vscode-webview://0jekke33biiooagsj8e6cokn7mg180focrdpc13ob6nks03d881e/utils/apktools.py#L40-L44) 里的 `beichen.jks`）：这是**汉化组自己的**密钥，用来给重打包后的 APK 签名，跟官方无关。

## 四、为什么官方这种"加密"形同虚设？

简单说就是**对称密码学滥用**：

1. 用 **PRNG（梅森旋转）** 当密钥流 —— MT19937 是为了统计随机性设计的，**不是密码学安全的 PRNG**，知道种子就能完全复现密钥流。
2. 种子又只是 **32 位 xxHash**，输入还是**公开的类名**。
3. 等于 `key = Hash(公开字符串)`，任何人都能算。

真正能拦住汉化的只有 SQLCipher 那一层（如果他们经常换密钥的话）。但开发商显然没把这部分当回事 —— 这也是为什么这个项目能纯粹靠"解包 → 改 → 重打包"路线完成汉化，而不需要做任何运行时 Hook。
